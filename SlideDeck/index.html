What Is A Design Pattern?

  • A "design pattern"
    • $6 phrase for "a way to organize your classes so they work together in a flexible way".
    • Like a for() loop, if(), array[]
    • Reusable technique, based around objects.
  • There are lots of design patterns:
    • Singleton, Factory, Template Method, Model-View-Controller, Observer, Dependency Injection, Facade, Finite State Machine,
    • and several more!
    • that you might NEVER use!
    • ...or that you might use all the time

What Is A Design Pattern?

  • Not platform specific.
  • Available for ColdFusion, Java, .NET, JavaScript, etc.
  • Most OOP langauges can do some variation of these.
  • Naming conventions and overall design is similar everywhere.

Using Design Patterns

  • Doesn't require an MVC framework
  • Doesn't always mean rewriting the entire codebase
    (MVC is the exception)
  • Don't have to use 1, some, or all the Design Patterns

They're not scary, I promise

Let's look at some patterns!

Singleton

  • Ensures that 1 and only 1 copy of a CFC gets created.
  • And that 1 copy is used by the entire application.
  • [Singleton Demo]

Strategy Pattern

  • For when you have classes that KIND OF behave the same way, but not exactly enough for Inheritance to be useful.
  • (Also a great intro to Interfaces)
  • Ducks are a great example!

Strategy Pattern and Ducks!

  • Ducks can fly, swim, quack, and eat
  • ALL ducks?
  • Rubber duck toys?
  • Wooden decoy ducks?
  • [Strategy Demo]

Strategy Pattern

  • Real world uses:
  • Payroll system: Full time employees, contractors, interns
  • Do they all get paid the same way?
  • Video games: things that fly...birds, insects, space ships
  • Do they all fly the same way?

Factory Pattern

  • Used for creating different variations of the same Object
  • You don't do createObject() directly
  • Ask the Factory to create an object for you
  • Requires only 1 CFC!
  • [Factory 1 Demo]
  • [Factory 2 Demo]

Adapter Pattern

  • Makes 2 (or more) CFCs that are different all run in the same way
  • You write 1 Adapter CFC to connect the components
  • Similar to "adapters" in the real world -- 3-prong adapter for a 2-prong power outlet, etc
  • [Adapter Demo]

Adapter Pattern

  • Real world use: swapping vendor libaries/APIs
  • Don't want to rebuild your entire app when switching vendors
  • Use an Adapter! Leave your app code as-is, just adapt the vendor methods so they work the same way

Decorator Pattern

  • Adds new behavior to a CFC
  • Replacemnet for tangled inheritance trees
  • Happens at run-time
  • Instead of inheriting the new functionality, the component is decorated with it

Decorator Pattern

  • Building an espresso machine
  • 4 types of coffee: House Blend, Dark Roast, Espresso, Decaf
  • Whip cream and/or hazelnut on all 4

Decorator Pattern

  • [Decorator Demo]

Decorator Pattern

  • Real world uses: any time you have a bunch of options, each with its own business logic
  • Adding logging, security, polling, and 2FA to a piece of functionality in your app
  • ...then later adding a feature flag requirement
  • ...write 1 FeatureFlag CFC, and decorate your app as needed.

Some Useful Tips

  • Don't feel like you have to jump in head first
  • Treat it like a tool in a tool box
  • Use obvious/standard naming conventions:
    SQLFactory, CustomerDecorator, etc
  • Dependency Injection is also very worth learning
    (it's not scary, I promise)

Recap

  • Design patterns...good!
  • Only needed a couple CFCs for each.
  • Able to combine patterns together for even more code reuse!
  • Did not require any 3rd party libaries.
  • But does take a little discipline and getting used to
